# MVPセキュリティ判断メモ

## 今回確定した範囲

- LINE Webhookは生のリクエスト本文と`LINE_CHANNEL_SECRET`からHMAC-SHA256署名を検証する。
- `LINE_CHANNEL_SECRET`未設定、署名なし、不正署名のWebhookは処理しない。
- backendの管理APIは`ADMIN_API_KEY`が設定され、`X-Admin-Key`と一致する場合だけ処理する。
- ブラウザはbackendを直接呼ばず、同一オリジンのNext.js管理プロキシを利用する。
- Next.jsサーバーは許可した管理APIパスだけを中継し、サーバー側で管理キーを付与する。
- CORSは`ADMIN_ORIGIN`の単一オリジンだけを許可し、未設定時はオリジンを許可しない。
- アプリケーションログへWebhook本文、LINEユーザーID、氏名、電話番号、応募・問い合わせ・送信本文を出力しない。

## RLSに関する未決定事項

Supabase Row Level Securityは今回のマイグレーションでは有効化しない。次の判断が未確定なためである。

- backendでservice roleを使用するか、利用者JWTをSupabaseへ伝播するか
- 1企業1環境で公開を開始するか、共有DBのマルチテナントで開始するか
- 採用担当者、企業管理者、サービス運用者の権限モデル
- 既存データをどの企業へ割り当てるか

RLSを有効化する前に、全業務テーブルの`company_id`適用とクエリ条件を完成させ、企業間アクセスの自動テストを用意する必要がある。

## 管理画面ログインに関する未決定事項

MVP公開時はNext.jsミドルウェアのBasic認証を暫定的な利用者認証とし、管理画面HTMLと`/api/admin/*`を同じ認証境界で保護する。資格情報は環境変数でのみ管理し、localStorage等へ保存しない。`ADMIN_API_KEY`は従来どおりNext.jsとbackend間だけで使用する。

`ADMIN_API_KEY`はブラウザからbackendを直接操作できないようにするサーバー間共有キーであり、利用者認証ではない。MVPの利用者認証は上記Basic認証が担当する。

正式公開には、Supabase Authなどの認証方式、セッション管理、招待・退職時の無効化、企業所属、役割別権限を決定して実装する必要がある。それまではアクセス制限された検証環境で運用する。

## マルチテナントに関する注意

マイグレーションは既存の主要テーブルへ`company_id`を追加し、既存行を`default`へ割り当てる。ただし現在の業務クエリすべてが企業条件を強制しているわけではないため、これだけでマルチテナント対応が完了するものではない。
