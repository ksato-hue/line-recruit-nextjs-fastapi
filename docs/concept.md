# LINE採用自動化サービス 構想

## 1. 本文書の役割

本文書は、サービスをなぜ作るのか、誰にどのような価値を届けるのか、将来どこへ発展させるのかを整理する。実装時の具体的な仕様と完了条件は[要件定義](requirements.md)を参照する。

元資料にない内容は断定せず、「案」または「要検討」として記載する。

## 2. サービスの目的

採用SNSで求人に興味を持った人が、応募前の不安をLINEで解消し、安心して次の一歩へ進める仕組みをつくる。

単に応募を急がせるのではなく、仕事内容、働き方、条件、選考などの疑問を受け止める。そのうえで、企業には問い合わせ対応、応募受付、面接調整を効率化する手段を提供する。

## 3. サービスが提供する価値

このサービスの核は、次の循環をつくることである。

1. 採用SNSで興味を持ってもらう
2. LINEで疑問や不安を解消する
3. 分かりやすい導線で応募・面接へ進んでもらう
4. 行動データを蓄積する
5. FAQ、求人情報、採用SNS、応募導線を改善する

高機能なマーケティングツールをそのまま採用へ持ち込むのではなく、採用に必要な導線へ絞ることで、導入と運用を簡単にする。

## 4. 採用企業側のメリット

- 繰り返し発生する質問をFAQで自動対応できる
- 応募者情報と問い合わせを一か所で確認できる
- LINE上で応募受付や面接候補日の調整ができる
- 少人数の採用体制でも対応漏れや初動の遅れを減らせる
- 応募前に多い不安や離脱箇所を把握し、採用コンテンツを改善できる
- 将来的には企業ごとの設定を管理画面で変更し、コード修正なしで運用できる

## 5. 応募者側のメリット

- 普段利用するLINEから気軽に質問できる
- 応募前に仕事内容や条件への不安を解消できる
- 応募方法や次の手順が分かりやすい
- LINE内で応募入力、内容確認、面接候補日の選択を進められる
- すぐに応募を決めていなくても、質問から関係を始められる

## 6. 想定する利用企業

- 中小から中堅規模で中途採用を行う企業
- Instagram、X、TikTokなどを採用広報に利用する企業
- SNSの閲覧や反応はあるが応募へ十分につながっていない企業
- 専任の採用担当者が少ない企業
- 大規模ATSやマーケティングオートメーションの導入・運用が重い企業

業種、採用人数、利用するSNS、既存ATSの有無による適合条件は要検討である。

## 7. 基本的な利用体験

### 7.1 応募者

```text
採用SNSを見る
  → LINE公式アカウントを追加する
  → FAQを見る、または自由に問い合わせる
  → 応募する
  → 内容を確認して確定する
  → LINEで面接候補日を受け取る
  → 希望日時を選ぶ
  → 確定通知を受け取る
```

このうち、友だち追加時の登録・あいさつと自動フォローは当初構想にあるが、現在の実装では未完成である。

### 7.2 採用担当者

```text
管理画面を開く
  → 新規応募、問い合わせ、対応状況を確認する
  → 応募者のメモやステータスを更新する
  → 必要に応じてLINEメッセージを送る
  → 面接候補日を登録して送信する
  → FAQ、質問ツリー、自動メッセージを調整する
  → 蓄積した数値から導線を改善する
```

管理画面認証、リマインド、詳細分析は今後の実装対象である。

## 8. サービスの差別化

- 採用SNSからLINE、応募前の疑問解消までに焦点を当てる
- 応募を強く促す前に、不安を受け止める体験を重視する
- FAQだけでなく、応募受付と面接調整まで一つの導線にまとめる構想を持つ
- 大規模ツールより設定項目を絞り、中小企業が運用しやすい形を目指す
- 行動データを採用SNSや求人内容の改善へ戻す

競合製品との機能・価格比較や、どの機能を独自価値とするかの検証は要検討である。

## 9. MVPの考え方

MVPでは、すべてを自動化することより、次の仮説を検証する。

- 応募前にFAQや問い合わせの場があると離脱を減らせるか
- LINE内の分かりやすい応募導線が応募完了につながるか
- 自動対応により採用担当者の工数を減らせるか
- FAQ閲覧や応募行動のデータが採用導線改善に役立つか

当初要件のMVPはFAQ、行動ログ、3日フォロー、応募計測、BIが中心である。構想メモではLINE内応募、面接調整、1時間・24時間リマインド、採用管理画面まで段階的にMVPへ含めている。

範囲が異なるため、まず検証したい仮説と実運用に必要な最低条件を決め、基準MVPを確定する必要がある。現在は面接調整や設定画面の一部が先行し、行動ログや自動フォローが未実装である。

## 10. 将来的な展開

元資料に記載された展開候補は次のとおりである。

- FAQにない質問へのAI回答
- AIが回答できない場合の有人問い合わせへの誘導
- 応募確度スコアリング
- セグメント配信
- 複数SNSの流入分析
- Googleカレンダーへの面接予定登録
- ATS連携
- 採用LPや採用SNSの改善提案
- 高度な採用向けマーケティングオートメーション
- 複数企業が設定だけで利用できるSaaS化

AI回答では、誤回答、採用条件の誤表示、個人情報入力、回答根拠の管理が課題になる。実装時期と安全基準は要検討である。

## 12. 設定画面とカスタマイズの構想

目標は、顧客ごとの変更でコード修正を必要とせず、管理画面から運用できる状態である。

想定される設定:

- 企業名、採用担当者、LINE Bot名
- 応募受付のON/OFF
- FAQ回答と公開状態
- 応募時の質問ツリー
- 応募完了、面接候補、面接確定、採否、リマインドのメッセージ
- 応募項目、必須・任意、表示順
- リマインド条件と送信タイミング
- 通知先と通知条件
- ステータスと選考フロー

現在はFAQ、限定的な質問ツリー、応募受付、一部メッセージ、会社情報の設定が実装されている。自由度を高めすぎると運用が複雑になるため、固定テンプレートと企業別設定の境界は要検討である。

## 13. データを資産として蓄積する考え方

蓄積する目的は、応募者を監視することではなく、応募前の不安と採用導線の課題を理解し、体験を改善することである。

価値が期待されるデータ:

- SNSや施策ごとのLINE登録数
- 閲覧されたFAQと未解決の質問
- 応募開始、応募完了、途中離脱
- ステータス変化と面接・採用への転換
- 自動フォロー前後の反応
- 問い合わせ分類と回答までの時間

企業はこのデータから、FAQ、求人票、採用SNS投稿、応募フォーム、フォロー内容を改善できる。サービス側で企業横断分析を行う場合は、利用目的、同意、匿名化、契約、個人情報保護を先に定める必要がある。

## 14. 今は実装しないアイデア

元資料上、初期MVPでは次を対象外とする。

- AI面接
- 複雑なシナリオ分岐
- 高度なスコアリング
- 複数チャネル配信
- ATSとの本格連携
- 高機能マーケティングツールと同等の細かな配信設計

また、次は将来構想として保持し、基準MVPの完了を妨げない範囲で検討する。

- Googleカレンダー連携
- AI FAQ
- 複数SNS横断分析
- 採用LP改善提案
- 契約・請求まで含むSaaS運用機能

## 15. 判断が未確定の項目

- 当初要件と拡張構想のどこまでを正式なMVPに含めるか
- 3日フォローと、応募途中1時間・24時間リマインドの優先順位
- LINE内応募を標準とするか、外部応募リンクも主要導線にするか
- ステータス体系を当初要件型と採用選考型のどちらに寄せるか
- 1企業1環境で開始するか、共有型マルチテナントで開始するか
- 認証基盤と企業・担当者の権限モデル
- 内蔵分析と外部BIの役割分担
- 応募者、LINEユーザー、応募履歴をどの粒度で分離するか
- FAQテンプレートを固定する範囲と、企業が自由追加できる範囲
- データ保存期間、削除、匿名化、企業横断分析のルール
- 価格、プラン、導入支援、保守範囲
- AI回答の品質基準、回答根拠、責任範囲
- 既存ATS、カレンダー、メールとの連携優先順位

## 16. 開発判断の原則

- 応募者が安心して判断できる体験を、強引な応募誘導より優先する
- 自動化によって採用担当者と応募者の双方の負担を減らす
- MVPでは検証する仮説に直接必要な機能を優先する
- 個人情報保護、企業データ分離、認証を後付け前提にしない
- 設定可能性と運用の分かりやすさのバランスを取る
- 行動データは改善目的で収集し、収集目的と利用範囲を明確にする
- 元資料にない事業・仕様判断は「案」として提示し、決定者の確認を得る
