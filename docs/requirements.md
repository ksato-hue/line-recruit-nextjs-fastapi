# LINE採用自動化システム 要件定義

## 1. 本文書の位置づけ

本文書は、当初の「採用LINE自動化 要件定義書 v0.1」、構想メモ、README、および現在の実装を基に、今後の開発判断に使用する要件を整理したものである。

資料と実装に差がある項目は、次の区分で記載する。

- **当初要件**: 要件定義書 v0.1 に明記された内容
- **拡張構想**: 構想メモに記載された段階的な機能拡張
- **現在の実装**: リポジトリ内で確認できる挙動
- **今後の方針**: 資料と実装の差を踏まえた方針。確定していない内容は「案」または「要検討」とする

## 2. プロジェクト概要

採用SNSから興味を持った応募検討者に対し、LINE公式アカウント上で応募前の疑問解消、応募受付、問い合わせ、面接日程調整を支援するシステムである。企業向けには、応募者情報や対応状況を管理するWeb管理画面を提供する。

高機能なマーケティングオートメーション製品全体を置き換えるのではなく、採用SNSから応募までの導線に特化した、導入しやすい軽量な自動化サービスとして開始する。

## 3. 解決したい課題

- SNSで求人に興味を持っても、疑問や不安を解消できず応募前に離脱する
- 採用担当者が同じ質問へ繰り返し回答し、個別対応の負担が大きい
- SNS、LINE、応募、面接の各段階が分断され、応募者の状況を追いにくい
- どのFAQや導線が応募につながったかを把握しにくい
- 少人数の採用体制では、応募途中のフォローや面接調整が遅れやすい

## 4. 対象ユーザー

### 4.1 導入企業

- 中小企業から中堅企業
- 中途採用を行う企業
- Instagram、X、TikTokなどで採用情報を発信する企業
- SNSから応募への転換に課題がある企業
- 採用担当者が少なく、問い合わせや日程調整の負担を減らしたい企業

### 4.2 利用者

- **応募者・応募検討者**: LINEで質問、応募、面接候補日の選択を行う
- **採用担当者**: 管理画面で応募者、問い合わせ、FAQ、面接候補、各種設定を管理する
- **運用管理者（将来）**: 導入企業、契約、権限、運用状況を管理する。詳細は要検討

## 5. システム構成

### 5.1 現在の構成

```text
応募者
  └─ LINE公式アカウント
       └─ LINE Messaging API / Webhook
            └─ FastAPI backend
                 ├─ Supabase PostgreSQL
                 └─ LINE Reply API / Push API

採用担当者
  └─ Next.js frontend
       ├─ FastAPI REST API
       └─ Next.js管理APIプロキシ（全管理API）

shared/faq_templates.json
  ├─ frontendのFAQ設定画面
  └─ backendのLINE FAQ応答
```

- `backend`: FastAPI。LINE Webhook、会話フロー、Supabase操作、管理APIを担当する
- `frontend`: Next.js。採用管理画面を担当する
- `shared`: FAQのカテゴリと質問テンプレートをJSONで共有する
- `Supabase`: 応募者、問い合わせ、LINE履歴、面接候補、FAQ・一般・質問ツリー設定の永続化先として使用する

### 5.2 当初要件との差

当初要件では初期APIをGASまたはNode.js、分析基盤をLooker StudioまたはMetabaseとする案だった。現在はFastAPIとNext.jsを採用しており、外部BI連携は実装されていない。

### 5.3 今後の方針

- 現在のFastAPI、Next.js、Supabase構成を当面の基準とする
- 認証方式はSupabase Authが当初案だが、正式決定は要検討
- BIは管理画面内の分析を拡張するか、外部BIを採用するか要検討

## 6. 機能要件

### 6.1 LINE応募受付

**当初要件**

- 友だち追加時にLINEユーザーを登録する
- あいさつ、FAQ、募集職種、応募前相談への導線を表示する
- 当初は応募リンクへの遷移とクリック記録を想定する

**拡張構想**

- 氏名、電話番号、希望職種、応募動機などをLINE内で入力する
- 応募内容の確認、修正、キャンセルをLINE内で完結する
- 将来は資格、希望勤務地などの項目も設定可能にする

**現在の実装**

- テキストメニューから応募を開始できる
- 氏名、電話番号、質問ツリーに基づく選択・自由回答を受け付ける
- 内容確認、修正、キャンセルを行える
- 応募確定時にSupabaseの`applicants`へ保存する
- 一般設定の`application_enabled`で応募受付を停止できる
- 友だち追加イベントの登録、流入元、応募リンククリックは未実装

**今後の方針**

- LINE内応募を現行MVPの標準導線とする案
- 外部応募フォームや求人ページへのリンクを併用するかは要検討
- 入力値の長さ、電話番号形式、必須条件を正式に定義する必要がある

### 6.2 応募途中の状態管理

**当初要件**

- `line_users`にステータス、最終行動、最終接触日時を保存する
- 「新規」「悩み中」「応募済」「対応中」「離脱」を管理する

**拡張構想**

- 応募開始後1時間と24時間でリマインドする
- 応募開始数、完了数、離脱数、離脱率を表示する

**現在の実装**

- 応募開始時に`application_sessions`へ企業・LINEユーザー単位のactiveセッションを作成し、既存activeセッションがあれば再開する
- 回答、現在の質問、最終操作日時を回答ごとにDBへ保存し、backend再起動後も続きから再開する
- 完了した応募者はSupabaseへ保存する
- プロセスメモリは応答生成用キャッシュであり、応募状態の正は`application_sessions`とする
- 同一WebhookイベントIDはセッションの`last_event_id`で直近処理を識別し、回答・完了の重複処理を防ぐ

**今後の方針**

- LINE送受信イベントから`last_contact_at`を更新・永続化し、管理画面の最終接触表示に使用する。実装までは応募者の`created_at`を「登録日時」として表示する
- ステータス体系は当初要件と構想メモで異なるため、業務ステータスと会話状態を分離して定義する（案）

#### 応募セッションのライフサイクル

1. 「応募する」で`active`セッションを作成する。同じ`company_id`・`line_user_id`のactiveセッションは部分一意制約により1件だけとし、存在する場合は再開する。
2. 回答ごとに`answers`、`current_question_key`、`last_activity_at`を更新する。`answers`には再開に必要な質問キーと回答だけを保持する。
3. 確認完了時はDBトランザクション内でセッション行をロックし、`applicants`を一度だけ作成してセッションを`completed`へ更新する。`completed_at`を設定し、個人情報の重複保持を避けるためセッションの`answers`を空配列にする。
4. 本人がキャンセルした場合は`cancelled`へ更新し、`cancelled_at`を設定して`answers`を空配列にする。

集計定義:

- **応募途中**: `application_sessions.status = active`。離脱状態を含む現在未完了の全セッション
- **離脱状態**: `status = active`かつ`last_activity_at`が判定時刻から24時間以上前。固定業務ステータスとして保存しない。判定時間は`APPLICATION_DROPOUT_HOURS`で変更可能
- **応募開始数**: `application_sessions`の全開始件数（active、completed、cancelledを含む）
- **応募完了数**: `application_sessions.status = completed`の件数
- **応募完了率**: 応募完了数 ÷ 応募開始数。開始数が0件の場合は算出しない

### 6.3 応募完了

- 応募内容を確認した後に確定できること
- 確定した応募者情報を永続化すること
- 応募完了メッセージを送信すること
- 二重送信時に重複登録しない仕組みは今後必要（要検討）
- 現在は応募完了メッセージを管理画面から変更できる

### 6.4 問い合わせ

- 応募者はLINEメニューから自由文の問い合わせを送信できること
- 問い合わせをLINEユーザーID、本文、日時と関連付けて保存すること
- 採用担当者は管理画面で問い合わせ一覧を確認できること
- 対応状況、担当者、返信、完了管理は未実装または要検討

### 6.5 FAQ

**当初要件**

- 仕事内容、給与、勤務地、応募条件、選考、休日、社風などの定型質問へボタンで回答する
- FAQクリックを行動ログとして記録する

**現在の実装**

- `shared/faq_templates.json`に17カテゴリ、248質問のテンプレートがある
- 管理画面で各質問の回答と表示ON/OFFを保存できる
- 回答があり表示ONのFAQのみLINEに表示する
- カテゴリと質問をページングして表示する
- 公開FAQがない場合は設定可能な準備中メッセージを返す
- 旧`faq_categories`/`faqs`方式のAPIと、新`faq_settings`方式が併存する
- FAQクリック専用の行動イベントは記録されない

**今後の方針**

- FAQの正規方式を`shared`テンプレート＋企業別回答設定とするか正式決定する
- 旧FAQ APIとテーブルの移行・廃止手順を定める
- FAQ閲覧イベントを企業、ユーザー、質問キーと関連付けて記録する

### 6.6 面接日程調整

**拡張構想**

- 採用担当者が応募者、面接回数、候補日を選びLINE送信する
- 応募者が候補日時を選択し、管理画面へ反映する

**現在の実装**

- 応募者詳細から面接種別と2〜5件の候補日時を登録できる
- 候補日時をLINE Push APIで送信できる
- 応募者が候補を選択し、確認後に確定できる
- 確定日時と面接ステータスをSupabaseへ反映する
- 面接履歴、変更・取消、競合防止、Googleカレンダー連携は未実装

### 6.7 管理画面

現在の管理画面は次を提供する。

- ダッシュボード集計
- 応募者一覧、検索、絞り込み
- 応募者詳細、ステータス、面接ステータス、メモ、タグの編集
- LINEメッセージ履歴と個別送信
- 問い合わせ一覧
- 面接候補日送信
- FAQ設定
- 質問ツリー設定
- 一般設定
- 簡易分析

ステータス、メッセージテンプレート、リマインド条件、質問ツリーは企業別に保存できる。リマインドの定期送信処理は未接続であり、管理者ログインと権限制御も未実装である。

### 6.8 各種設定

現在は企業名、採用担当者名、LINE Bot表示名、通知先メール、応募受付ON/OFF、および一部自動メッセージを設定できる。

今後の候補:

- 採用・不採用通知テンプレート
- リマインド時刻、送信条件、送信文面
- 応募フォーム項目と必須条件
- ステータス定義
- LINEチャネル情報
- 営業時間、通知条件、担当者権限

元資料に詳細がない項目は、実装前に仕様決定を必要とする。

### 6.9 質問ツリー

- 管理画面で最初の質問、選択肢、選択肢ごとの追加質問を設定できる
- backendは企業IDに対応する設定を読み、LINE応募フローへ反映する
- 設定がない場合は希望職種と応募動機を尋ねる既定ツリーを使用する
- 現在は質問配列として、テキスト、電話番号、複数行テキスト、選択式、必須設定、並び順を管理する
- 旧分岐形式は読み取り時に新形式へ変換し、明示的に保存するまではDBを上書きしない

### 6.10 リマインド

**当初要件**は「悩み中かつ最終接触から3日経過し、応募済みではないユーザー」へのフォローである。**拡張構想**は応募途中ユーザーへの1時間後・24時間後の再通知である。対象、目的、時刻が異なるため、別のリマインド種別として扱う案とする。

自動送信ジョブは未接続である。`application_sessions`には最終操作日時と1時間・24時間・3日通知の送信日時を持たせ、将来「activeかつ所定時間以上操作なし、対応する送信日時が未設定」を対象として抽出できる。誤解を避けるため、設定画面のリマインドON/OFF・時間・テンプレート選択は既存値を保持したまま無効化する。

### 6.11 通知

- 現在はLINE Reply APIとPush APIによるメッセージ送信がある
- 通知先メールアドレスを保存できるが、メール通知処理は未実装
- 採用・不採用通知は構想にあるが未実装
- 送信失敗時の再試行、失敗通知、配信停止、送信時間帯は要検討

### 6.12 行動ログ・分析

- 当初要件ではFAQクリック、応募リンク、メッセージ、自動フォロー、人対応、ステータス変更を記録する
- 現在は主にLINE送受信を`line_message_logs`へ記録する
- ダッシュボードは応募者と問い合わせの現在値から件数を集計する
- 流入元、FAQ閲覧、応募開始、フォロー効果、ステータス変更履歴は未実装
- 外部BI連携は未実装

## 7. 非機能要件

### 7.1 セキュリティ

- LINE Webhookは署名を検証し、LINEからの正当なリクエストだけを処理すること
- 管理画面は認証を必須とし、企業と役割に応じて認可すること
- 読み取り・更新を含む管理APIを認証・認可の対象とすること
- 管理APIキーなどの秘密情報はブラウザへ公開しないこと
- 本番環境では秘密情報未設定時に保護が無効になるfail-open動作を禁止すること
- CORSは許可する管理画面オリジンへ限定すること
- SupabaseはRLSまたは同等の制御で企業間のアクセスを分離すること
- 入力検証、レート制限、依存関係の脆弱性管理を行うこと

現在はLINE Webhook署名検証と、全管理APIに対するfail-closedの共有キー認証を実装している。共有キーはNext.jsサーバーからのみ付与し、ブラウザへ公開しない。ただし、この共有キーは採用担当者本人を識別するログイン機能ではなく、利用者認証・役割別認可は未実装である。

### 7.2 個人情報保護

- LINEユーザーID、氏名、電話番号、応募内容、問い合わせ、面接情報を個人情報またはそれに準じる情報として扱う
- 利用目的、保存期間、削除方法、閲覧権限を定めること
- ログへリクエスト本文や個人情報を無制限に出力しないこと
- 開発・検証環境へ実在応募者データを流用しないこと
- 外販時に特定企業のデータ、設定、ノウハウをそのまま転用しないこと

保存期間、削除依頼手順、バックアップ保持期間は要検討である。

### 7.3 可用性

- 会話途中状態はプロセス再起動や複数インスタンスでも継続できること
- LINEとDBの一時障害時に、タイムアウト、再試行、失敗記録を行うこと
- 同一Webhookの再送や管理操作の再実行に対して冪等性を確保すること
- バックアップ、復旧目標、監視・アラート基準は要検討

### 7.4 ログ

- エラー、外部API呼び出し、Webhook処理結果、管理操作を追跡できること
- 構造化ログを使用し、企業IDや処理IDで追跡できること（案）
- 個人情報、アクセストークン、管理キーを平文で記録しないこと
- FAQ閲覧やステータス変更など、分析用イベントと障害調査用ログを分離すること

### 7.5 マルチテナント

- すべての企業別データを`company_id`に関連付けること
- DBクエリ、管理画面、LINEチャネル、設定、ログ、ジョブで企業境界を強制すること
- 現在の`COMPANY_ID`環境変数は設定系の一部にしか適用されず、完全なマルチテナントではない
- 1企業1環境で開始するか、初期から共有環境にするかは要検討

### 7.6 保守性・拡張性・コスト

- FAQ、質問、メッセージをコード変更なしで更新できる範囲を増やす
- DB変更はマイグレーションとして管理する
- 小規模企業が導入可能な運用コストを目指す
- 将来のAI応答、スコアリング、複数チャネル、ATS連携を追加できる境界を維持する

## 8. データ設計の方針

### 8.1 当初のデータモデル案

- `companies`: 導入企業
- `line_users`: LINE友だち、流入元、状態、最終接触
- `user_events`: FAQ、応募、フォローなどの行動ログ
- `messages`: メッセージテンプレート
- `message_deliveries`: 配信履歴
- `applications`: 応募行動と応募状態

### 8.2 現在コードから確認できるデータ

- `applicants`: 応募者と選考状態
- `inquiries`: 問い合わせ
- `interview_slots`: 面接候補と確定状態
- `line_message_logs`: LINE送受信履歴
- `faq_categories` / `faqs`: 旧FAQ方式
- `faq_settings`: FAQテンプレートに対する企業別回答・公開設定
- `app_settings`: 企業別のキー・値設定
- `question_tree_settings`: 企業別質問ツリー
- `application_sessions`: LINE応募の開始、回答進捗、完了・取消、最終操作、将来のリマインド送信判定

### 8.3 今後の方針

- 実際のSupabaseスキーマをマイグレーションとしてリポジトリ管理する
- 全企業別テーブルへ`company_id`、作成日時、更新日時を持たせる
- `line_user_id`単独ではなく企業との組み合わせで一意性を定義する
- 応募者、LINEユーザー、応募レコードを統合するか分離するか要検討
- イベントログは追記型とし、集計に必要な時系列を失わない設計とする（案）
- 設定は型付きカラムまたは検証可能なJSONを優先し、任意のキー・値だけに依存しない（案）
- ステータス値、外部ID、参照整合性、削除方針をDB制約へ反映する

## 9. 環境変数

### 9.1 現在使用している変数

| 変数 | 使用箇所 | 用途 | 備考 |
|---|---|---|---|
| `SUPABASE_URL` | backend | Supabase URL | 必須 |
| `SUPABASE_KEY` | backend | Supabase APIキー | anon/service roleの使い分けを明文化する必要がある |
| `LINE_ACCESS_TOKEN` | backend | LINE Messaging API | 必須 |
| `LINE_CHANNEL_SECRET` | backend | LINE Webhook署名検証 | 必須。未設定時はWebhookを拒否 |
| `ADMIN_ORIGIN` | backend | CORSで許可する管理画面オリジン | 未設定時はオリジンを許可しない |
| `COMPANY_ID` | backend | 現在の企業識別子 | 既存データは`default`を前提とする |
| `ADMIN_API_KEY` | backend / frontend server | 全管理APIのサーバー間共有キー | 必須。`NEXT_PUBLIC_`を付けない |
| `BACKEND_API_BASE_URL` | frontend server | 管理APIプロキシのbackend URL | 必須。ブラウザはbackendを直接呼ばない |

### 9.2 今後の方針

- frontend用とbackend用の環境変数サンプルを実装と一致させる
- 必須変数は起動時に検証し、欠落時は明確に失敗させる
- 秘密情報を`NEXT_PUBLIC_`変数へ格納しない
- 認証、ジョブ、メール、監視を採用する場合の変数は方式決定後に追加する

## 10. MVPの範囲

### 10.1 基準MVP

元の要件定義書を基準とする場合、次をMVP完了対象とする。

- LINE友だち登録とあいさつ導線
- FAQボタン応答
- 応募または応募導線
- 問い合わせ受付
- 行動ログ
- ステータス管理
- 3日未接触フォロー
- 応募リンクまたは応募完了の計測
- 基本指標の可視化
- 個人情報保護、認証、企業データ分離の最低限の安全対策

### 10.2 拡張MVP候補

構想メモに基づく次の機能を基準MVPへ含めるかは要検討である。

- LINE内の可変応募フォーム
- 応募途中1時間・24時間リマインド
- 面接候補送信と確定
- 合否通知
- 詳細な採用管理画面
- 一般設定、質問ツリー、テンプレート管理

現在はこの拡張MVP候補の一部が、基準MVPの未実装機能より先に実装されている。

## 14. 既知の課題

- 会話状態がインメモリで、再起動と水平スケールに対応できない
- 管理API共有キーは利用者ログインの代替にならず、Next.js管理プロキシ自体への利用者認証がない
- LINE/DB操作にトランザクション、冪等性、十分なタイムアウト・再試行がない
- FAQの新旧データ方式が併存する
- 企業IDが全データへ適用されていない
- RLSと全業務クエリの企業スコープ強制が未実装である
- 現在固定しているNext.js依存関係にnpm auditの既知脆弱性があり、安全な更新にはメジャーバージョン移行検証が必要である

## 15. 完了条件

### 15.1 各機能の共通完了条件

- 正常系、入力エラー、外部サービス障害時の挙動が定義されている
- 企業境界と権限が守られる
- 必要なデータが永続化され、管理画面とLINEで整合する
- 重複イベントや再試行で不正な二重処理が起きない
- 個人情報や秘密情報がログへ漏れない
- 自動テストまたは再現可能な検証手順がある
- DB変更、環境変数、運用方法が文書化されている

### 15.2 基準MVPの完了条件

- LINE登録からFAQ、問い合わせ、応募までの主要導線を本番相当環境で完遂できる
- 応募途中と最終接触を永続化できる
- 採用担当者が認証済み管理画面で応募者と問い合わせを確認できる
- 対象条件どおりにフォローを一度だけ送信できる
- LINE登録、FAQ閲覧、応募開始・完了、フォロー送信を計測できる
- 企業別データ分離、Webhook署名検証、秘密情報管理が実装されている
- 障害監視、バックアップ、基本的な運用手順が定義されている

## 16. 用語定義

| 用語 | 定義 |
|---|---|
| 応募者 | 応募を開始または完了し、採用管理の対象となるLINE利用者 |
| 応募検討者 | 求人に関心があるが、応募を完了していないLINE利用者 |
| 採用担当者 | 導入企業側で応募者対応と設定管理を行う利用者 |
| FAQ | 応募前によくある質問と回答 |
| 質問ツリー | LINE応募で表示する最初の質問、選択肢、追加質問の構造 |
| 会話状態 | LINE上で現在どの入力・確認段階にいるかを示す一時状態 |
| 業務ステータス | 新規応募、面接調整中、採用など採用業務上の進行状態 |
| 行動イベント | FAQ閲覧、応募開始、送信など利用者の時系列行動記録 |
| リマインド | 条件と経過時間に基づく自動フォローメッセージ |
| マルチテナント | 複数企業が同一サービスを利用しつつ、データと設定が分離される構成 |
| MVP | 価値と運用可能性を検証するための最小限の製品範囲 |
